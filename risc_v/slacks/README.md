
# Слаки  

**Slack** - запас времени для определённого маршрута в схеме.  

Есть два вида слаков:  
**Setup slack** = Required_Time - Arrival_Time  
**Hold slack** = Arrival_Time - Required_Time  

**Required_Time** - требуемое время прибытия сигнала данных для корректной предустановки/удержания.  
**Arrival_Time** - реальное время прибытия сигнала данных.  

В соответствии с этими определениями, можно записать следующее:  
- **Setup slack** > 0 - сигнал придёт раньше, чем необходимо, время предустановки не нарушено.
- **Setup slack** = 0 - сигнал пришёл именно в момент времени предустановки.
- **Setup slack** < 0 - сигнал опоздал и нарушил время предустановки.  
- **Hold slack** > 0 - сигнал закончился позже, чем время удержания. Время удержания не нарушено.
- **Hold slack** = 0 - сигнал закончился именно в момент времени удержания.  
- **Hold slack** < 0 - сигнал закончился слишком рано и нарушил время удержания.  
# Причины нарушения setup/hold slack  

## Слишком много уровней логики  

Если между двумя последовательностными элементами находится сложная многоуровневая логика, то сигнал может просто не успеть дойти до второго последовательностного элемента и нарушиться время предустановки.  
  
**Основные причины:**  
- сложные условия  
- нетривиальные математические операции с широкими данными  
**Методы борьбы:**  
- Переписывание кода(разбиение условий и выражений на более простые)  
- Конвееризация - сохранение промежуточных результатов в триггер, что позволит разбить общий маршрут на два или более простых. Но добавится задержка в несколько тактов в зависимости от количества промежуточных триггеров.    
- Ретайминг - оптимизация синтезатора, заключающаяся в автоматическом переставлении триггеров в схеме для минимизации периода тактовой частоты.  
> Но синтезатор имеет ряд ограничений на автоматическую балансировку регистров и триггеров. Во первых, синтезатор применяет ретайминг только для критического пути или для мест, где схема не выдерживает ограничений требуемой частоты. Во вторых, в случае, если в коде самому наставить несколько триггеров подряд в надежде на то, что синтезатор применит ретайминг, синтезатор может просто объединить их в регистр и больше не применять никаких оптимизаций.  

## Большой fanout  

Fanout триггера - последовательностные элементы, принимающие один и тот же сигнал с этого триггера.  

**Основная проблема:**  
- На этапе синтеза проблемы нет. Проблема возникает на этапе размещения элементов на кристалле, поскольку сигнал может физически не успеть дойти до всех триггеров за время равное периоду тактового сигнала и опять возникнет setup slack violation.  
**Методы борьбы:**  
- Репликация - создание второго такого же триггера и разделение fanout между ними.  

## Упаковка элементов  

В fpga есть два типа памяти - распределённая(distributed ram) и блочная(block ram).  

**Distributed RAM:**  
- Построена на соединённых между собой LUT-ах без дополнительной аппаратуры.  
- Подходит, только если требуется очень небольшой объём памяти, иначе используется очень много логических блоков просто на реализацию памяти.  

**Block RAM(BRAM):**  
- Полноценная двухпортовая(как правило) память.  
- Подходит если требуется использовать большой объём памяти, чтобы не использовать для этого логические блоки.  

Также в fpga существует блок **DSP** - Digital Signal Processing для осуществления быстрых арифметических операций как например умножение или аккумулирующее сложение. Без этого блока пришлось бы потратить значительное количество логических блоков на реализацию специализированного умножителя.  

**Упаковка** - процесс замещения регистров или триггеров, описанных в RTL-коде на регистры или триггеры внутри **BRAM** или **DSP**.  
> Да, BRAM и DSP имеют фиксированное внутреннее устройство, но если их входные или выходные регистры совпадают с описанием в RTL, то синтезатор попробует не размещать их в LUT и напрямую направить их входные сигналы внутрь блоков BRAM и DSP.  

Существует проблема упаковки элементов в эти блоки синтезатором при осуществлении оптимизаций. Поскольку для борьбы со слаками часто требуется вводить дополнительные триггеры для балансировки сложной логики, то нужно внимательно следить за тем, чтобы они не были заменены синтезатором на триггеры из **BRAM** или **DSP**.
